# i18n Conventions (fundamental-ngx)

## Architecture

Signal-based internationalization. No async pipes needed.

```
FD_LANGUAGE_SIGNAL  (WritableSignal<FdLanguage>)  -- source of truth
        |
        |  derives via linkedSignal
        v
FD_LOCALE_SIGNAL    (WritableSignal<string>)      -- formatting locale
        |
        v
UI5 Bridge          (calls setLanguage())         -- opt-in via provideUi5LanguageBridge()
```

**Key principle:** Set `FD_LANGUAGE_SIGNAL` and let everything else derive. Only touch `FD_LOCALE_SIGNAL` for the rare case where formatting locale must differ from the language.

## API Patterns

```typescript
// Single translation
protected readonly _label = resolveTranslationSignal('coreButton.label');

// Multiple translations via factory (same access modifier for factory + results)
private readonly _translate = resolveTranslationSignalFn();
private readonly _submitLabel = this._translate('coreForm.submit');
private readonly _cancelLabel = this._translate('coreForm.cancel');

// Global language switch -- locale and UI5 follow automatically
this.langSignal.set(FD_LANGUAGE_GERMAN);
```

## Language Objects

Each `FdLanguage` carries `locale` and `name` metadata alongside translation keys. The `locale` field drives `FD_LOCALE_SIGNAL` via `linkedSignal`.

## Browser Auto-Detection

Enabled by default via `FD_LANGUAGE_AUTO_DETECT`. Resolution: exact locale match, then base language, then English fallback.

## Adding / Renaming / Removing Keys

The only manual file is **`libs/i18n/src/lib/models/fd-language.ts`** — update the `FdLanguage` interface first, then use the CLI (`nx run i18n:i18n-manage --command=add|rename|remove`). The `/i18n-manage` skill has the full workflow.

Skipping `fd-language.ts` causes `TS2554: Expected 2 arguments, but got 1` — see the skill's Troubleshooting section.
