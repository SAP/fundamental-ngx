UI5 Web Components Scoping Learn how to use the UI5 Web Components scoping feature to avoid naming conflicts when multiple versions coexist in your application or when building micro-frontends.

What is Scoping?

Scoping is an optional feature that lets you add a custom suffix to all UI5 Web Components' custom element names. This is particularly useful when:

  • Building Libraries: Prevent naming conflicts with host applications
  • Micro-Frontends: Allow multiple independent applications to coexist
  • Version Compatibility: Use different versions of UI5 Web Components simultaneously
  • Third-Party Integration: Ensure your components work alongside other libraries

Setup: Configuring Scoping

Important: Scoping configuration must be done before importing any UI5 Web Components. The configuration is applied at the module level when components are imported.

How Scoping Works with Angular Wrappers

Why Angular Wrappers are Scoping-Compatible

The Angular wrapper components are fully compatible with UI5 Web Components scoping:

  • Flexible Selectors: Components use both element and attribute selectors (e.g., ui5-avatar, [ui5-avatar]), so they work with any tag name variation
  • Native Element Reference: The wrapper gets the actual custom element via elementRef.nativeElement, which works regardless of scoping
  • Template Projection: Components use <ng-content> for projection, making them agnostic to the underlying tag names
  • Property Synchronization: Input/output synchronization works on the actual DOM element, regardless of its tag name

Example Component Structure

Angular wrapper components use both element and attribute selectors to support scoped web components:

  • Element selector: ui5-avatar or ui5-avatar-myapp (when scoped)
  • Attribute selector: [ui5-avatar] works with any element name
  • Template uses ng-content for projection, agnostic to tag names
  • Property/event synchronization works on the actual DOM element

Best Practices
  • Configure Early: Set up scoping in a separate configuration file imported before any components
  • Use Unique Suffixes: Choose meaningful, unique suffixes to identify your library (e.g., organization name)
  • Document Your Suffix: Make it clear in your library's documentation which suffix is used
  • Scoping Rules: Use fine-grained scoping rules to exclude custom components if needed
  • Version Management: Combine scoping with careful dependency versioning for complex micro-frontend setups
