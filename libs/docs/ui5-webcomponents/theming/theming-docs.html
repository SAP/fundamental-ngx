Overview

The UI5 theming bridge keeps fundamental-ngx and UI5 Web Components in sync — when you switch themes (light/dark, high-contrast), both sides update together automatically.

Setup

Three steps are required:

Step 1: Register UI5 theming assets in main.ts (before Angular bootstrap):

// main.ts
import '@ui5/webcomponents-theming/dist/Assets.js';

This registers the CSS variable sets for all SAP themes. Without it, setTheme() calls inside UI5 won't find the theme data. The assets are loaded lazily on first use — no upfront bundle size impact.

Step 2: Add the theming bridge and core theming to your app providers:

// app.config.ts
import { provideTheming, themingInitializer } from '@fundamental-ngx/core/theming';
import { provideUi5ThemingBridge } from '@fundamental-ngx/ui5-webcomponents-base/theming-bridge';

export const appConfig: ApplicationConfig = {
  providers: [
    provideTheming({ defaultTheme: 'sap_horizon' }),
    themingInitializer(),
    provideUi5ThemingBridge(),
  ]
};

Step 3: Instantiate the theming provider for each UI5 package you use:

// app.config.ts (continued)
import { inject, provideEnvironmentInitializer } from '@angular/core';
import { Ui5WebcomponentsMainThemingService } from '@fundamental-ngx/ui5-webcomponents/theming';

export const appConfig: ApplicationConfig = {
  providers: [
    // ... steps 1-2 above ...
    provideEnvironmentInitializer(() => inject(Ui5WebcomponentsMainThemingService)),
  ]
};

Each UI5 package (@ui5/webcomponents, @ui5/webcomponents-fiori, @ui5/webcomponents-ai) ships its own theming provider that registers theme assets for that package's components. Angular's tree-shaking removes providedIn: 'root' services that are never injected, so you must explicitly instantiate each provider your app needs.

Without Step 3, the bridge has no registered providers to call — theme switches will have no effect on UI5 components.

Multiple UI5 packages

If your app uses components from multiple UI5 packages, instantiate a provider for each one:

import { Ui5WebcomponentsMainThemingService } from '@fundamental-ngx/ui5-webcomponents/theming';
import { Ui5WebcomponentsFioriThemingService } from '@fundamental-ngx/ui5-webcomponents-fiori/theming';
import { Ui5WebcomponentsAiThemingService } from '@fundamental-ngx/ui5-webcomponents-ai/theming';

providers: [
  provideUi5ThemingBridge(),
  provideEnvironmentInitializer(() => inject(Ui5WebcomponentsMainThemingService)),
  provideEnvironmentInitializer(() => inject(Ui5WebcomponentsFioriThemingService)),
  provideEnvironmentInitializer(() => inject(Ui5WebcomponentsAiThemingService)),
]

Only include the packages you actually use — each provider lazily imports its own theme JSON, so unused providers won't add to your bundle.

How it works

The data flow:

ThemingService.currentTheme   (Observable, set by provideTheming + init)
        |
        |  Ui5ThemingBridgeService subscribes
        v
Ui5ThemingService.setTheme()  (calls each registered provider — shadow DOM re-renders)

Ui5ThemingBridgeService is activated by provideUi5ThemingBridge(). It subscribes to the core ThemingService.currentTheme Observable and calls Ui5ThemingService.setTheme() whenever the theme changes. The Ui5ThemingService then delegates to all registered providers (one per UI5 package).

The bridge handles both initial theme (from URL ?theme= parameter or config default) and runtime theme switches. It uses takeUntilDestroyed() for automatic cleanup.

Apps not using UI5 components don't need this bridge at all — it lives in its own entrypoint (@fundamental-ngx/ui5-webcomponents-base/theming-bridge) so the core @fundamental-ngx/core dependency is never pulled in unless you explicitly import it.
