Common Issues and Solutions

This section covers common problems you might encounter when implementing i18n in your application, along with their solutions. Use this guide to quickly diagnose and fix issues.

🔴 Problem: Translations don't update when I change the language Symptom: You change the language using FD_LANGUAGE_SIGNAL.set(), but the UI still shows the old language.

Common Causes and Solutions:

❌ Cause 1: Not updating a WritableSignal

You provided a signal() but you're trying to change it without getting a reference to the writable signal.

Wrong: 
// In providers
{{ '{' }}{{ '{' }} provide: FD_LANGUAGE_SIGNAL, useValue: signal(FD_LANGUAGE_ENGLISH) {{ '}' }}{{ '}' }}

// In component - can't modify this!
private readonly lang = inject(FD_LANGUAGE_SIGNAL); // Signal<FdLanguage> (read-only)
this.lang.set(newLang); // ❌ Error: set doesn't exist
 Correct: 
// In providers
{{ '{' }}{{ '{' }} provide: FD_LANGUAGE_SIGNAL, useValue: signal(FD_LANGUAGE_ENGLISH) {{ '}' }}{{ '}' }}

// In component - cast to WritableSignal
private readonly lang = inject(FD_LANGUAGE_SIGNAL) as WritableSignal<FdLanguage>;
this.lang.set(newLang); // ✅ Works!
❌ Cause 2: Provider scope issue

The language signal is provided in a component, but you're trying to change it from a different component that doesn't share the same provider.

Solution: Provide the language signal at a higher level (app.config.ts or shared parent component) so all components can access the same signal instance. 
// app.config.ts
export const appConfig: ApplicationConfig = {{ '{' }}{{ '{' }}
  providers: [
    {{ '{' }}{{ '{' }}
      provide: FD_LANGUAGE_SIGNAL,
      useValue: signal(FD_LANGUAGE_ENGLISH)
    {{ '}' }}{{ '}' }}
  ]
{{ '}' }}{{ '}' }};
❌ Cause 3: Not calling the signal in templates

When using fdTranslate pipe or resolveTranslationSignal(), you forgot to call the signal with ().

Wrong: 
<div>{{ '{' }}{{ '{' }} ('myKey' | fdTranslate) {{ '}' }}{{ '}' }}</div>  <!-- Shows [Function] -->
<div>{{ '{' }}{{ '{' }} myLabel {{ '}' }}{{ '}' }}</div>  <!-- Shows [Function] -->
 Correct: 
<div>{{ '{' }}{{ '{' }} ('myKey' | fdTranslate)() {{ '}' }}{{ '}' }}</div>  <!-- ✅ Shows translated text -->
<div>{{ '{' }}{{ '{' }} myLabel() {{ '}' }}{{ '}' }}</div>  <!-- ✅ Shows translated text -->
🔴 Problem: I see English text instead of my target language Symptom: You've set up a language (e.g., Spanish, French), but the UI shows English translations instead.

Common Causes and Solutions:

❌ Cause 1: Language not imported

You're referencing a language constant without importing it.

Wrong: 
// No import for FD_LANGUAGE_SPANISH
{{ '{' }}{{ '{' }} provide: FD_LANGUAGE_SIGNAL, useValue: signal(FD_LANGUAGE_SPANISH) {{ '}' }}{{ '}' }}
 Correct: 
import {{ '{' }} FD_LANGUAGE_SPANISH {{ '}' }} from '@fundamental-ngx/i18n';

{{ '{' }}{{ '{' }} provide: FD_LANGUAGE_SIGNAL, useValue: signal(FD_LANGUAGE_SPANISH) {{ '}' }}{{ '}' }}
❌ Cause 2: Translation key doesn't exist in target language

The translation key exists in English but not in your custom language. The system falls back to English.

Solution: When creating custom languages, always extend an existing language to inherit missing keys: 
import {{ '{' }} FD_LANGUAGE_ENGLISH {{ '}' }} from '@fundamental-ngx/i18n';

const MY_CUSTOM_LANGUAGE: FdLanguage = {{ '{' }}{{ '{' }}
  ...FD_LANGUAGE_ENGLISH,  // ✅ Inherit all English translations as fallback
  coreButton: {{ '{' }}{{ '{' }}
    ...FD_LANGUAGE_ENGLISH.coreButton,
    close: 'Cerrar'  // Override just this key
  {{ '}' }}{{ '}' }}
{{ '}' }}{{ '}' }};
❌ Cause 3: Using wrong translation key

The translation key you're using doesn't exist in any language.

Solution: Check the available keys by looking at the FdLanguage interface or examining FD_LANGUAGE_ENGLISH object structure. Use TypeScript autocomplete to find valid keys.
🔴 Problem: Parameters in translations don't work (showing {{ '{' }}paramName{{ '}' }} literally) Symptom: Your translation shows {{ '{' }}count{{ '}' }} or {{ '{' }}name{{ '}' }} as literal text instead of the actual values.

Common Causes and Solutions:

❌ Cause 1: Using double curly braces instead of single

ICU Message Format uses single curly braces {{ '{' }}param{{ '}' }}, not double.

Wrong: 
// In your language object
counterMessage: 'You have {{ '{' }}{{ '{' }} count {{ '}' }}{{ '}' }} items'  // ❌ Double braces
 Correct: 
// In your language object
counterMessage: 'You have {{ '{' }}count{{ '}' }} items'  // ✅ Single braces
❌ Cause 2: Not passing parameters when resolving

You defined parameters in the translation string but didn't pass them when resolving.

Wrong: 
// Translation: "Hello {{ '{' }}name{{ '}' }}"
const greeting = resolveTranslationSignal('myApp.greeting');  // ❌ Missing parameters
 Correct: 
// Translation: "Hello {{ '{' }}name{{ '}' }}"
const greeting = resolveTranslationSignal('myApp.greeting', {{ '{' }} name: 'John' {{ '}' }});  // ✅ Pass parameters
❌ Cause 3: Parameter name mismatch

The parameter name in the translation doesn't match what you're passing.

Wrong: 
// Translation: "You have {{ '{' }}count{{ '}' }} items"
const message = resolveTranslationSignal('myApp.items', {{ '{' }} total: 5 {{ '}' }});  // ❌ 'total' !== 'count'
 Correct: 
// Translation: "You have {{ '{' }}count{{ '}' }} items"
const message = resolveTranslationSignal('myApp.items', {{ '{' }} count: 5 {{ '}' }});  // ✅ Matches parameter name
❌ Cause 4: Spaces inside parameter braces

ICU syntax requires no spaces: {{ '{' }}count{{ '}' }} not {{ '{' }} count {{ '}' }}.

Wrong: 
counterMessage: 'You have {{ '{' }} count {{ '}' }} items'  // ❌ Spaces around 'count'
 Correct: 
counterMessage: 'You have {{ '{' }}count{{ '}' }} items'  // ✅ No spaces
🔴 Problem: TypeScript errors with translation keys Symptom: You get TypeScript errors like "Type 'string' is not assignable to type 'FdLanguageKeyIdentifier'".

Common Causes and Solutions:

❌ Cause 1: Using invalid translation key

The key you're using doesn't exist in the FdLanguage interface.

Solution: Use TypeScript autocomplete to find valid keys. Type the component or feature name to see available keys: 
// Type 'coreButton.' and let autocomplete show you valid keys
resolveTranslationSignal('coreButton.close');  // ✅ Valid

// Invalid key - TypeScript will catch this
resolveTranslationSignal('coreButton.invalid');  // ❌ TypeScript error
❌ Cause 2: Key stored in a variable without proper typing

When storing keys in variables, TypeScript can't verify they're valid.

Wrong: 
const myKey = 'coreButton.close';  // Type: string
resolveTranslationSignal(myKey);  // ❌ TypeScript error
 Correct (Option 1): Use 'as const' 
const myKey = 'coreButton.close' as const;  // Type: 'coreButton.close'
resolveTranslationSignal(myKey);  // ✅ Works
 Correct (Option 2): Type as FdLanguageKeyIdentifier 
import {{ '{' }} FdLanguageKeyIdentifier {{ '}' }} from '@fundamental-ngx/i18n';

const myKey: FdLanguageKeyIdentifier = 'coreButton.close';
resolveTranslationSignal(myKey);  // ✅ Works
 Correct (Option 3): Inline the key 
resolveTranslationSignal('coreButton.close');  // ✅ Always works
🔴 Problem: "inject() must be called from an injection context" Symptom: Error when calling resolveTranslationSignal() or resolveTranslationSignalFn().

Common Causes and Solutions:

❌ Cause 1: Calling in a method

resolveTranslationSignal() can only be called during initialization, not inside methods.

Wrong: 
class MyComponent {{ '{' }}{{ '{' }}
  onClick(): void {{ '{' }}{{ '{' }}
    // ❌ Error: not in injection context
    const label = resolveTranslationSignal('myApp.label');
  {{ '}' }}{{ '}' }}
{{ '}' }}{{ '}' }}
 Correct: Use resolveTranslationSignalFn() instead 
class MyComponent {{ '{' }}{{ '{' }}
  // ✅ Get the resolver function in injection context
  private readonly getTranslation = resolveTranslationSignalFn();

  onClick(): void {{ '{' }}{{ '{' }}
    // ✅ Call it in the method
    const label = this.getTranslation('myApp.label');
    console.log(label());
  {{ '}' }}{{ '}' }}
{{ '}' }}{{ '}' }}
❌ Cause 2: Calling in a callback or async context

Functions like resolveTranslationSignal() must be called during initialization, not in callbacks or async contexts.

Wrong: 
import {{ '{' }} afterNextRender {{ '}' }} from '@angular/core';

constructor() {{ '{' }}{{ '{' }}
  afterNextRender(() => {{ '{' }}{{ '{' }}
    // ❌ Error: not in injection context
    const label = resolveTranslationSignal('myApp.label');
  {{ '}' }}{{ '}' }});
{{ '}' }}{{ '}' }}
 Correct: Create signal during initialization 
import {{ '{' }} effect, afterNextRender {{ '}' }} from '@angular/core';

// ✅ Create the signal as a class field (in injection context)
private readonly label = resolveTranslationSignal('myApp.label');

constructor() {{ '{' }}{{ '{' }}
  // ✅ Use effect() to react to signal changes
  effect(() => {{ '{' }}{{ '{' }}
    console.log('Label:', this.label());
  {{ '}' }}{{ '}' }});

  // ✅ Use afterNextRender() for DOM-related operations
  afterNextRender(() => {{ '{' }}{{ '{' }}
    // Signal was already created, just use it
    document.title = this.label();
  {{ '}' }}{{ '}' }});
{{ '}' }}{{ '}' }}

// ✅ Or use in any method/callback
onClick() {{ '{' }}{{ '{' }}
  alert(this.label());  // ✅ Works - signal created at initialization
{{ '}' }}{{ '}' }}
❌ Cause 3: Calling outside a component/service class

These functions require Angular's dependency injection.

Solution: Use the TranslationResolver class directly for standalone functions or pass the translation as a parameter from a component.
🟡 Problem: Performance issues with translations Symptom: Your app feels slow or you notice excessive change detection cycles.

Best Practices for Performance:

✅ Use signals, not Observables

The signal-based API (FD_LANGUAGE_SIGNAL) is more efficient than the deprecated Observable-based API (FD_LANGUAGE) because it uses a pure pipe instead of an impure pipe, resulting in significantly fewer change detection cycles. 

// ✅ Modern signal-based (fast)
{{ '{' }}{{ '{' }} provide: FD_LANGUAGE_SIGNAL, useValue: signal(FD_LANGUAGE_ENGLISH) {{ '}' }}{{ '}' }}

// ❌ Deprecated Observable-based (slow)
{{ '{' }}{{ '{' }} provide: FD_LANGUAGE, useValue: of(FD_LANGUAGE_ENGLISH) {{ '}' }}{{ '}' }}
✅ Reuse translation signals

Create the signal once as a class field, don't recreate it multiple times.

Wrong (inefficient): 
<div>{{ '{' }}{{ '{' }} ('myApp.label' | fdTranslate)() {{ '}' }}{{ '}' }}</div>
<div>{{ '{' }}{{ '{' }} ('myApp.label' | fdTranslate)() {{ '}' }}{{ '}' }}</div>
<div>{{ '{' }}{{ '{' }} ('myApp.label' | fdTranslate)() {{ '}' }}{{ '}' }}</div>
 Correct (efficient): 
// In TypeScript
readonly label = resolveTranslationSignal('myApp.label');

// In template
<div>{{ '{' }}{{ '{' }} label() {{ '}' }}{{ '}' }}</div>
<div>{{ '{' }}{{ '{' }} label() {{ '}' }}{{ '}' }}</div>
<div>{{ '{' }}{{ '{' }} label() {{ '}' }}{{ '}' }}</div>
✅ Avoid calling TranslationResolver repeatedly

If you must use the TranslationResolver class directly, cache the result instead of calling resolve() repeatedly.

✅ Quick Troubleshooting Checklist

Use this checklist to quickly diagnose common i18n issues:

Before asking for help, check:

  • Are you using FD_LANGUAGE_SIGNAL (not the deprecated FD_LANGUAGE)?
  • Did you cast to WritableSignal<FdLanguage> when you need to change the language?
  • Are you calling signals with () in your templates?
  • Did you import the language constant you're using (e.g., FD_LANGUAGE_SPANISH)?
  • Does your custom language extend an existing language with ...FD_LANGUAGE_ENGLISH?
  • Are you using single curly braces {{ '{' }}param{{ '}' }} in ICU syntax (not double)?
  • Are there no spaces inside parameter braces?
  • Are you passing the correct parameter object when resolving translations?
  • Are translation keys valid according to FdLanguageKeyIdentifier?
  • Are you using resolveTranslationSignalFn() for method-based translations?
  • Is the language signal provided at the right level (app-wide or component-specific)?
