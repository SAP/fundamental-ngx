Comparison: When to use what approach

There are multiple ways to use translations in fundamental-ngx. This comparison table helps you choose the right approach for your situation.

Approach Best For Key Info
fdTranslate pipe
Simple inline translations in templates (button labels, headings, one-off text) Very clean syntax, no TypeScript needed. Can't reuse across components or use in logic.
resolveTranslationSignal()
Reusable translations, TypeScript code, computed values, class field initializers Reusable signal, works in templates and code. Must call in injection context (not inside methods).
resolveTranslationSignalFn() Translations inside methods, dynamic keys, callbacks, event handlers Works outside injection context and in methods. Slightly more complex (two-step process).
patchLanguage()
Override specific translations for one component, A/B testing, custom wording Component-scoped overrides. Not for general translation use, requires understanding providers.
TranslationResolver
Low-level control, one-time synchronous lookups, non-reactive scenarios Full control but requires manual signal management. More boilerplate, easy to use incorrectly.

Quick Decision Guide:

Start here if you're unsure:

  1. In templates for simple text? → Use fdTranslate pipe
  2. Need it in TypeScript code or reusable? → Use resolveTranslationSignal()
  3. Need it inside a method? → Use resolveTranslationSignalFn()
  4. Want to override specific translations? → Use patchLanguage()
  5. Need low-level control? → Use TranslationResolver class

Using TranslationResolver class

The TranslationResolver class is the low-level API for resolving translation keys to their translated strings. It provides fine-grained control over translation resolution.

When to use TranslationResolver:

  • When you need direct control over translation resolution
  • For one-time synchronous translation lookups (non-reactive)
  • In advanced scenarios where utility functions don't fit

Note: For most use cases, prefer the simpler resolveTranslationSignal() utility function (shown in the next section) which handles signal management automatically.

Using utility functions (Recommended)

The recommended approach for most use cases is to use the signal-based utility functions. These functions provide a simpler API while automatically handling signal management and cleanup.

Available utility functions:

  • resolveTranslationSignal(key) - Use in injection context to get a reactive signal for a translation key
  • resolveTranslationSignalFn() - Returns a resolver function that can be used outside injection context

These functions must be called within the dependency injection context (constructor, class field initializers, or factories).

Using fdTranslate pipe

The fdTranslate pipe provides a convenient way to resolve translations directly in templates. The pipe returns a Signal<string>, so you must invoke it with () in your template.

When to use the pipe:

  • For simple inline translations in templates
  • For one-off labels and text that don't need to be reused
  • When you want a clean, declarative syntax

When NOT to use the pipe:

  • If you need to reuse the same translation in multiple places (use resolveTranslationSignal())
  • For computed values based on translations (use computed() with signals)
  • In class methods or logic (use utility functions)

Important: The pipe is pure for better performance, meaning it only re-evaluates when the language signal changes.
