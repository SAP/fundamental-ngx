Changing languages at runtime

You can change languages at runtime by calling langSignal.set(newLanguage) on the FD_LANGUAGE_SIGNAL token. When you change the language, all UI elements—labels, buttons, messages, and counters—update automatically. The locale (FD_LOCALE_SIGNAL) and UI5 component language also follow automatically — one call is all you need.

The example below demonstrates switching between English, Ukrainian, and a custom language. Watch how all translations update instantly when you select a different language. This works because Angular's signal-based reactivity automatically propagates changes to all dependent translations, whether they're created with resolveTranslationSignal(), the fdTranslate pipe, or computed() signals.

Key concepts: Language controls UI text translations (like button labels and messages), while locale controls date and number formatting. Locale auto-derives from the language's locale field via linkedSignal — you no longer need to set both manually. You can also provide custom translation objects that implement the FdLanguage interface—useful for localization variants or overriding specific translations.

In a real application, provide FD_LANGUAGE_SIGNAL at the root level (typically in app.config.ts) so language changes affect your entire app. This example shows it at the component level for demonstration purposes only.

Note: Use FD_LANGUAGE_SIGNAL (the modern signal-based approach). The old FD_LANGUAGE Observable token is deprecated and will be removed in a future version.

Browser auto-detection

By default, FD_LANGUAGE_SIGNAL auto-detects the user's language from Angular's LOCALE_ID at bootstrap. This means your app starts in the user's preferred language without any explicit setup.

The detection resolves the browser locale against all 37 built-in languages: exact match first, then base language fallback (e.g., pt-BR resolves to Portuguese), then English as the ultimate fallback.

To disable auto-detection, provide {{ '{' }} provide: FD_LANGUAGE_AUTO_DETECT, useValue: false {{ '}' }} in your app config.

Locale override (edge case)

FD_LOCALE_SIGNAL normally derives from the language's locale field via linkedSignal. For the rare case where you need different formatting than the language implies (e.g., German UI text with Japanese date formatting), you can override it explicitly with localeSignal.set('ja-JP').

Important: The override is contextual — when the language changes next, linkedSignal resets the locale to the new language's default. This prevents stale locale overrides from persisting across language switches.

When to set what — quick reference
Scenario What to do
Default (95% of apps) Nothing extra — language auto-detects from the browser, locale derives automatically, UI5 follows if the bridge is active.
Explicit language at startup Provide FD_LANGUAGE_SIGNAL with signal(FD_LANGUAGE_*) in app.config.ts.
Switch language at runtime Call langSignal.set(FD_LANGUAGE_*) — locale and UI5 follow automatically.
Different formatting than language implies Call localeSignal.set('ja-JP') after setting the language. Resets on next language change (linkedSignal behavior).
Disable auto-detection Provide {{ '{' }} provide: FD_LANGUAGE_AUTO_DETECT, useValue: false {{ '}' }}.
UI5 web components Add provideUi5LanguageBridge() to your app providers.

Key principle: Set FD_LANGUAGE_SIGNAL and let everything else derive. Only touch FD_LOCALE_SIGNAL for the rare case where the formatting locale must differ from the language.

UI5 Web Components integration

Apps using @fundamental-ngx/ui5-webcomponents can activate the language bridge via provideUi5LanguageBridge() from @fundamental-ngx/ui5-webcomponents-base/i18n. This ensures UI5 web components react to fundamental-ngx language changes by calling UI5's setLanguage() reactively.

Setup: Add provideUi5LanguageBridge() to your application providers (in app.config.ts). It uses provideEnvironmentInitializer to inject Ui5LanguageService, which watches FD_LOCALE_SIGNAL via effect() and calls setLanguage() whenever the locale changes.

Result: Changing FD_LANGUAGE_SIGNAL updates fundamental-ngx translations, the formatting locale, and UI5 component language — all with one call.

Changing locales at runtime

Translations have a concept of locale, which is a combination of language and region. For example, English language has different locales for US, UK, Australia, etc. The locale controls date, number, and currency formatting as well as pluralization rules.

Note: In most cases you do not need to set the locale manually — it auto-derives from the language via linkedSignal. This example demonstrates direct locale control for cases where you need to explore formatting differences independently of language.

To control the locale directly, inject FD_LOCALE_SIGNAL and call .set(). The locale should be in the format of language-region (e.g., en-US or uk-UA).

Note: The old FD_LOCALE (Observable) token is deprecated. Use FD_LOCALE_SIGNAL instead.

Creating custom translations

You can create your own custom language objects by implementing the FdLanguage interface. This is useful for localization variants (e.g., US English vs UK English), testing with mock translations, or providing domain-specific terminology for your application.

Translation values can be either strings with interpolation using single curly brackets ({{ '{' }}paramName{{ '}' }}) for simple parameter substitution, or functions that receive a parameters object for complex logic like conditional pluralization or dynamic formatting.

Translation Approaches:

  • Built-in (English): Standard translations provided by fundamental-ngx. Simple and ready to use.
  • String Interpolation: Uses template strings like "{{ '{' }}count{{ '}' }} characters remaining". Best for simple parameter substitution. Values are automatically inserted at runtime.
  • Function-based: Uses functions like (params) => `${{ '{' }}params.count{{ '}' }} items`. Best for conditional logic, complex pluralization, or when you need full JavaScript control. Notice how the counter messages in the example below show warnings when count is low!

To find available translation keys for each component, check the component's documentation—the I18n section lists all translation keys with their default English values. You only need to provide translations for the keys you want to customize; missing keys will fall back to empty strings.
